The WhatsApp phone-link flow now uses the same Meta env-variable contract in both local and live. Local development still skips the real outbound Meta HTTP request, but it now calls the same verification-template sender adapter with a no-op provider transport so missing `META_WHATSAPP_TOKEN`, `META_PHONE_NUMBER_ID`, `META_WHATSAPP_VERIFY_TEMPLATE_NAME`, `META_WHATSAPP_VERIFY_TEMPLATE_LANGUAGE_CODE`, or `META_API_VERSION` will fail locally too. That keeps the local configuration surface aligned with live while preserving the local debug verification-link loop.

The authenticated account-summary contract now includes `email_contact_methods` and `phone_contact_methods` arrays built from `stephen_dcx_users_contact_methods`, with phone rows also carrying the preferred linked provider channel. The app account surface was rebuilt around that normalized model: top primary email/phone summaries are now read-only, verified badges are shown for both confirmed email and verified phone, and the lower sections list the full email and phone contact methods instead of pretending the primary rows are casual inline-edit fields.

The phone-number management UI now lives in the `Phone numbers` section. Existing numbers show badges and an edit action, and a local/dev successful send exposes the debug verification link right in that section so the Meta click can be simulated without leaving the page. The WhatsApp verification page was also made friendlier for the already-consumed-success case so a user who is already signed in does not get stranded on a harsh invalid-token screen after the phone has actually been verified.

One intentional gap remains before claiming full multi-contact management: authenticated additional-email add/edit/verify flows do not exist yet as a safe backend capability. The UI now shows the `Emails` section and the read-only list of normalized email contact methods, but the `Add email` button is still intentionally disabled with copy explaining that authenticated email management needs its own verification flow. Phone add/edit is real; email list display is real; authenticated multi-email mutation is the next boundary if we decide to build it before live.
