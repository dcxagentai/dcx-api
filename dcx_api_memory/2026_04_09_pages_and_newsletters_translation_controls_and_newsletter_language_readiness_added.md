Pages and newsletters now expose real translation mechanics on the admin side instead of being merely translation-ready in schema. The page detail capability and newsletter detail capability were expanded to return translation summaries with existing live rows and missing supported languages. Two new creation capabilities and routes were added so admins can create a missing page translation or newsletter translation directly from the current source route and then navigate into the new localized editor path immediately.

Newsletter detail now also returns recipient language-readiness metadata. It groups announcement-eligible, confirmed users by preferred language, checks whether a live newsletter translation exists for that language, and reports both ready-to-send users and users blocked by missing translations. This powers the client-facing readiness warning in admin and aligns with the explicit policy decision for newsletters: do not silently fall back to English when a user prefers another language.

The newsletter send-preparation capability was updated to match that policy. It still creates prepared send rows, recipient snapshots, and tracked-link rows, but users whose preferred language lacks a live newsletter translation are now marked as skipped with a missing-translation reason instead of being counted as send candidates. Targeted send-preparation tests, new translation-create tests, and the full backend app suite all passed after this change.
